00 — What we're working with

The box arrives ready.
We just have to find it.

The hardware ships with everything we need already installed. Our job is not to set up a computer — it's to get a secure connection to one that's already running, then deploy our software remotely.

The key fact

Ubuntu is already installed.

The appliance ships with a customized Ubuntu 24.04 LTS (ARM64) pre-configured by the manufacturer. CUDA, Docker, the NVIDIA container toolkit, PyTorch, and all AI development tooling are present from first boot. We do not install an operating system. We do not flash firmware. We do not prepare installation media.

All we need to do is turn it on, get it on the network, and establish a secure tunnel back to our office. Everything after that happens from our keyboards.

What ships with the box

Pre-installed

  • Ubuntu 24.04 LTS (ARM64, customized)
  • GPU drivers and CUDA toolkit
  • Docker with GPU support
  • Management dashboard (GUI)
  • SSH key and tunnel tooling
What we add remotely

Our deployment

  • Mesh VPN agent (Tailscale)
  • Application stack (Docker Compose)
  • Local AI model weights
  • Database schema and configuration
  • Monitoring and backup agents
What stays on-site

Customer's side

  • The physical appliance
  • Their existing file storage (NAS)
  • Network infrastructure
  • Monitor/keyboard for first boot only
  • Nothing else to maintain
Phase 0 — recommended
Pre-stage

Configure before shipping

2-3 days · Our facility

The ideal flow: the appliance ships to us first, we unbox and fully configure it, then forward it to the customer. They receive a plug-and-play device. This eliminates most risk from the customer-site phase. Skip this phase only if timing requires direct-to-customer shipping.

0.1

Receive and validate hardware

In-office~30 min

Open the box, inspect for shipping damage, verify SKU matches order (4 TB NVMe variant, 128 GB RAM). Confirm accessories: power adapter, documentation, any included cables. Photograph the unit's serial number and NVIDIA asset tag for our records.

0.2

First boot and account creation

In-office~20 min

Connect display via HDMI 2.1 or USB-C DisplayPort. Connect keyboard and Ethernet. Power on. Walk through the Ubuntu OEM setup wizard:

  • Language: English (US)
  • Keyboard layout: US
  • Network: Ethernet (DHCP from our office router)
  • User account: admin with strong passphrase (stored in our password manager)
  • Computer name: sb-[customer-code] — we use a stable identifier that survives the move
  • Disk encryption: enabled with passphrase stored in our vault
  • Auto-login: disabled
0.3

Install the VPN mesh agent

In-office~5 min

Install Tailscale and authenticate the device to our tailnet using a pre-generated auth key. This is the single most important step — it's what lets us reach the box from anywhere in the world without any network configuration on the customer's side.

Terminal
$ curl -fsSL https://tailscale.com/install.sh | sh $ sudo tailscale up \ --authkey=tskey-auth-XXXXXXXX \ --ssh \ --hostname=sb-customer-code # Verify connectivity from a second machine on the tailnet $ ssh admin@sb-customer-code
0.4

Deploy the application stack

In-office~2 hrs

From our laptop, SSH into the appliance and run our deployment script. This pulls the Docker images, downloads the AI model weights, initializes PostgreSQL with pgvector, and brings up all services. End-to-end validation with a test video.

  • Clone our deployment repository
  • Populate environment variables (customer-specific settings)
  • Run docker compose up -d
  • Wait for all services to pass health checks
  • Drop a test video in the watched folder, confirm end-to-end processing
  • Perform sample search queries, confirm expected results
0.5

Take a snapshot, then ship

In-office~1 hr

Create a full system snapshot as our known-good recovery point. Export a backup of the database and configuration. Power down cleanly, package with original materials, ship to customer with tracking. Include our printed quick-start sheet for the customer's physical setup.

Note on direct-to-customer shipping

If the timeline doesn't allow pre-staging, skip Phase 0 and start with Phase 1. The appliance still ships with Ubuntu pre-installed by the manufacturer. The difference is we'll do steps 0.3 and 0.4 over a video call with someone at the customer site, rather than in our office. Add approximately 1 hour to the on-site time in that scenario.

Phase 1 — on the customer site
On-site

Physical setup, guided

~30 min · Video call

The only time the customer needs to touch the appliance. We schedule a video call, walk them through unpacking and cabling, and stay on the line until the box is on their network. No technical skill required — our printed sheet covers every step with photos.

1.1

Physical unboxing and connection

Customer~10 min

On a scheduled video call, we walk the customer through unpacking and cabling. Our printed quick-start sheet goes in the shipping box, so they have a reference even if the call drops.

  • Display: HDMI 2.1 cable to any monitor (or laptop with HDMI input)
  • Keyboard: USB-C keyboard (or USB-A via adapter)
  • Network: Ethernet cable from router or switch to the RJ45 port
  • Power: Included 240W adapter to standard US outlet
  • Press the power button on the front panel
1.2

First boot wizard — if not pre-staged

Customer + us~15 min

Only applies if we skipped Phase 0. The customer shares their screen on the video call. We dictate each selection; they click. This is mechanical — no judgment required on their part.

  • Walk through the language, keyboard, and time zone selections
  • Connect to their Wi-Fi or confirm Ethernet auto-detection
  • Create the admin account with our pre-agreed credentials
  • Set the computer name per our naming convention
  • Skip optional cloud account integrations
  • Let the system complete its initial setup and reach the desktop
1.3

Open a terminal, nothing else

Customer~2 min

The customer opens a terminal from the application launcher. We dictate a single command for them to paste and run. They don't need to understand it — they just need to copy and paste.

Phase 2 — establishing the connection
Critical handoff

Ten minutes that matter

~10 min · The handoff

This is the pivotal moment. Once Tailscale is running and we have SSH access, the customer can walk away. Everything that follows happens from our desks. Get this right and the rest is trivial. Get it wrong and we're stuck.

2.1

Install the VPN agent

Customer types~3 min

We generate a fresh single-use authentication key from our Tailscale admin console immediately before the call. We dictate or share the command; the customer pastes and runs.

What the customer runs
# Download and install $ curl -fsSL https://tailscale.com/install.sh | sh # Authenticate this device to our mesh $ sudo tailscale up \ --authkey=tskey-auth-XXXXXXXX \ --ssh \ --hostname=sb-customer-code # Confirm it registered (they should see an IP address) $ tailscale status
Why Tailscale specifically

It traverses NAT and firewalls automatically. No port forwarding on the customer's router. No static IP required. No coordination with their IT team. The mesh is WireGuard-based (modern, fast, secure), uses identity-based authentication, and gives the device a stable address even if its local network changes. Free for up to twenty devices.

2.2

Verify from our side

Our office~1 min

From our laptop, SSH into the appliance using its Tailscale hostname. Tailscale SSH handles authentication automatically using our team identities — no password prompt, no separate key exchange.

What we run
$ ssh admin@sb-customer-code.tail-xyz.ts.net # You should see the MOTD and a shell prompt # Welcome to Ubuntu 24.04 LTS (DGX OS) # System information as of ... admin@sb-customer-code:~$ _

Take a screenshot and share it with the team. This moment — first successful SSH — is when the project genuinely begins.

2.3

Release the customer

Customer~1 min

Thank them. Confirm we have everything we need. Let them know the next contact will be when we have something to show them (usually 1-2 business days). They can unplug the monitor and keyboard if they want — the appliance will keep running headless from now on. They should leave it powered on and connected to the network.

Phase 3 — the remote work
Remote

Everything else, from our desks

~4 hours · Our office

With SSH in hand, this phase is just deployment. The customer is not involved. We execute our standard deployment sequence, validate each service, and prepare the system for the customer's content.

3.1

System hardening

SSH~45 min

Baseline security posture before anything else goes on the machine.

  • Apply all pending system updates
  • Set the system time zone to match the customer's location
  • Enable the firewall with Tailscale as the only permitted ingress
  • Install and configure Fail2ban for SSH brute-force protection
  • Enable automatic security updates for critical packages
  • Install monitoring agent with alerts routed to our team
3.2

Application stack deployment

Docker~2 hrs

The main work. This is the sequence that was rehearsed in Phase 0 (if pre-staged) or done fresh (if direct-to-customer).

  • Pull our deployment repository (private, via deploy key)
  • Generate environment configuration with customer-specific values
  • docker compose up -d brings up all services
  • AI model weights download from our mirror (approximately 15 GB)
  • Initialize PostgreSQL schema with pgvector extension enabled
  • Seed the settings table with local provider configuration
  • Run health checks on every service endpoint
ARM64 note

A few Python libraries don't ship ARM64 wheels and need to compile from source. This can add up to 20 minutes during initial deployment. Our Docker images bake these in, so the customer's first deploy doesn't hit this — it's already handled during image build.

3.3

Connect to the customer's storage

Mount~30 min

Where the customer keeps their content lives outside the appliance. We mount it read-only (or read-write for the watched folder) over the local network.

  • Identify the customer's NAS or file server (IP, credentials, share path)
  • Install the appropriate client (NFS or CIFS/SMB)
  • Mount the share at /mnt/customer-storage
  • Add the mount to /etc/fstab for auto-mount on reboot
  • Test read access from the application container
  • Configure the watched folder that triggers ingestion
3.4

Validate end-to-end

Us~30 min

Before we invite the customer back in, we prove the full pipeline works with their environment. Drop a few test files into the watched folder, monitor the logs as they flow through transcription, visual analysis, embedding, and indexing. Run sample queries against the resulting knowledge base.

3.5

Bulk indexing kickoff

Overnight2-3 days

Point the ingestion job at the customer's existing archive. Depending on volume, this runs for 2-3 days in the background. We monitor progress remotely; nothing is required from the customer during this time.

Phase 4 — delivery and training
Handover

Show them what they own

~2 hours · Video call

Once the initial indexing is complete, we schedule a second video call to demonstrate the working system, train their team, and hand over documentation. Brando leads this call; we stay available for technical questions.

4.1

Live demonstration

Show the search interface with their actual content indexed. Run the kinds of queries they'll actually use. Demonstrate that results include the original file references so they can click through to the source. Let them try queries themselves — this is the first moment they see it working on their data.

4.2

Team training

Walk through the interface with each team member who will use it. Show them how to add content, how to phrase queries for best results, how to share findings with colleagues. This typically takes 30 minutes per person. Record the session for their future reference.

4.3

Operational runbook

Deliver a printed and digital runbook covering the things a non-technical person might need: how to tell if the system is healthy, who to call if something looks wrong, how to pause indexing, how to add a new user. Keep it under 15 pages with screenshots.

4.4

Support handoff

Establish the support channel — dedicated Slack shared with their team, email alias, emergency phone number. Set expectations for response time. Confirm our monitoring is routing their system's alerts to us, not to them.

06 — If something goes wrong

What breaks glass looks like.

Four scenarios we plan for. In three years of doing this, we've hit roughly one of them per five deployments. The point isn't that things never go sideways — it's that we have a named response for each case.

Scenario
Response
Tailscale won't install or connect
Fall back to a commercial remote desktop tool — AnyDesk or TeamViewer — installed manually during the video call. Requires someone to be present at the customer site whenever we work, so it's slower, but it reaches the same end state. We keep the installer cached on our ops machine, ready to email as an attachment.
Customer's network blocks everything
We ship a cellular backup kit with the appliance — an LTE USB modem and a prepaid SIM. The runbook includes instructions for the customer to plug it in. The appliance then has its own internet connection, independent of their network. Used only when their IT environment is uncooperative.
Hardware fault before or during deployment
The manufacturer operates a US-domestic RMA program with overnight swap. We restore the new unit from our pre-shipping snapshot, which takes about 90 minutes. The customer loses roughly a business day; they do not need to do anything other than accept the replacement delivery.
Something we can't diagnose remotely
Brando handles on-the-ground logistics in the United States and can reach the customer's site for any issue that demands physical presence. This is our failsafe and has been used exactly once — a power outlet the customer had miswired.
07 — What we need from you

Before the box ships.

A short checklist for Brando to gather from the customer ahead of deployment. Most of this is information we need to configure the appliance during Phase 0 — if we have it in advance, the customer-site time drops to the absolute minimum.

Shipping

Where and when

  • Delivery address with recipient name
  • Earliest date they can accept delivery
  • Whether someone will be present to receive
  • Any loading dock or access constraints
Network

Connectivity details

  • Where the appliance will physically sit
  • Whether Ethernet is available at that location
  • IT contact name and phone (in case we need them)
  • Any known network restrictions (air-gapped, special VLAN)
Storage

What the appliance will read

  • Existing NAS or file server IP and share path
  • Credentials or read-only account we can use
  • Approximate size of the archive to index
  • File formats present (video codecs, document types)
Users

Who will use the system

  • Names and email addresses of initial users
  • Any preferred authentication method (SSO, local accounts)
  • Training session preferred date and attendees
  • Administrative contact on their side going forward
